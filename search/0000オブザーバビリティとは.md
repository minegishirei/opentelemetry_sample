## オブザーバビリティとは?

オブザーバビリティとは、システムの内部状態を外部から観測可能にするための概念です。
ログ、メトリクス、トレースなどのテレメトリデータを活用し、システムの挙動や問題の根本原因を迅速に特定できるようにします。
これにより、運用者は障害対応やパフォーマンス改善を効率的に行うことが可能となります。

オブザーバビリティで重要なのは**透明度**です。
「事前に予測していた起こり得るエラーをキャッチできる能力」ではなく
「 **いまだに起きたことがないエラーを含めキャッチすることができ、何が起きているかを把握できる能力** 」といえます。

- フロントエンドのレイヤー
- バックエンドのレイヤー
- インフラレイヤー

これらのレイヤーを包括的に確認し、「想定外のエラー」にも対応するのがオブザーバビリティの試みです。



## できること/できないこと

### オブザーバビリティでできること

* **障害の原因特定（Root Cause Analysis）の高速化**  
    * ログ・メトリクス・トレースを横断的に関連付けることで、  
    「どのリクエストが」「どのコンポーネントで」「なぜ失敗したのか」を追跡できます。  
    特に分散システムでは、単一のログ監視では不可能な可視化が可能です。
* **未知の障害・想定外の挙動の発見**  
    * 事前にアラート条件を定義していなくても、  
    レイテンシの異常分布やエラーパターンの変化などから  
    「何かがおかしい」ことに気づけます。
* **システム全体の振る舞いの理解** 
    * ツールによっては、単一のサービスではなく、**エンドツーエンドの流れ**を把握できます。
      * フロントエンド → バックエンド
      * API → バッチ
      * アプリケーション → インフラ 
* **パフォーマンスボトルネックの特定**  
    * CPUやメモリといったインフラ指標だけでなく、  
    アプリケーション内部の処理時間や外部依存（DB / 外部API）まで含めて分析できます。
   


### オブザーバビリティでできないこと

- インフラやアプリケーションの自動修復や自動スケーリングの直接的な実行。サービスの構成変更やデプロイの自動化。NewRelic内のエラーを検知し原因を特定することは得意ですが、挙動を直接変更することは向いていません。
- 障害の完全な予防。可視化はできますが、すべての障害を未然に防ぐ仕組みではありません。設計ミスや外部要因による障害は発生します。
- セキュリティ脆弱性の自動検出や修正。ツールによっては、ある程度の脆弱性診断は可能ですが、侵入検知等は範囲外のケースが多いです。
脆弱性スキャンや侵入検知（IDS / IPS）そのものを代替するものではありません。
- ビジネスインテリジェンス。プロダクトのKPI/コンバージョン率の測定を目的としてはいません。
